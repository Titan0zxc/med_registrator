## 1. Предметная область и терминология
- **Цифровой двойник (Digital Twin)** — не просто программа, а модель, которая имитирует поведение реального сотрудника колл-центра. Он принимает решения на основе правил (как живой оператор).
- **Медицинский регистратор** — сотрудник, который распределяет поток пациентов: оценивает срочность, назначает время, направляет к нужному врачу.
- **Триаж (Triage)**  — система сортировки пациентов по тяжести состояния (Красный → Жёлтый → Зелёный → Синий). Это твоя ключевая логика.
- **Полис ОМС/ДМС** — идентификатор пациента в медицинской системе (МИС). Через него программа проверяет, зарегистрирован ли человек.
- **Талон** — выходной документ (время, врач, кабинет, номер очереди).
- **API МИС** — внешняя система, где хранятся: регистратура, расписание врачей, история болезней.
- **API страховой компании** — туда после приёма уходит информация о факте обслуживания (для оплаты).
- **Платёжная система (опционально)** — если услуга платная или штраф за неявку.

## 2. 4 сценария Use Case

## **Сценарий 1:** Заполнение анкеты (входные данные)
  * Актёр: Пациент.
  * Что делает система: Принимает текст + структурированные поля.
  * Пример данных:
  * Возраст: 65
  * Температура: 38.5
  * Что беспокоит: Кашель, боль в груди
  * Хронические: Гипертония
  * Адрес: ул. Ленина, 5
## **Сценарий 2:** Проверка (идентификация + МИС)
  * Что делает система: Отправляет запрос в API МИС:
  * «Пациент с полисом XXX существует?»
  * «Его прикреплённая поликлиника?» (сюда записывать по умолчанию).
  * «Был ли он на приёме на прошлой неделе?» (чтобы не спамил).
  * Фильтр: Если пациента нет в базе — выдать ошибку «Обратитесь в регистратуру лично».
## **Сценарий 3:** Определение тяжести состояния (триаж)
  * Это сердце твоей программы. Превращает симптомы в приоритет.
  * Красный (скорая):
  * Условие: Температура > 39.5 ИЛИ «боль в груди» ИЛИ «потеря сознания» ИЛИ возраст < 3 с температурой > 39.
  * Действие: Не давать талон! → Вызвать API скорой помощи (Сценарий 4).
  * Жёлтый (срочно, сегодня):
  * Условие: Температура 38-39.5, сильный кашель, рвота.
  * Действие: Запись в «кабинет неотложной помощи» на сегодня.
  * Зелёный (планово):
  * Условие: Температуры нет, симптомы лёгкие (насморк).
  * Действие: Запись к терапевту на любой день.
  * Синий (консультация / диспансер):
  * Условие: Жалоб нет, нужна справка / рецепт.
  * Действие: Запись через 3 дня.
## **Сценарий 4:** Реагирование и выдача талона
  * Для Зелёного/Жёлтого/Синего:
  * Запрос в API расписания: «Дай ближайший слот в поликлинике по адресу X».
  * Бронируем слот.
  * Формируем Талон (время, кабинет, врач, номер очереди).
  * Дополнительно: Отправляем в личный кабинет «активную запись» (где можно отменить).
  * Для Красного:
  * Отправляем JSON в API скорой: {"address": "...", "symptoms": "...", "priority": "CRITICAL"}.
  * Пациенту: «Вызов передан, бригада едет. Оставайтесь на месте».
  
## 3. Каналы и фильтры:
- Фильтры (последовательная обработка):
- Фильтр синтаксиса: Заполнены ли все поля? Возраст — число?
- Фильтр идентификации: Есть ли пациент в API МИС?
- Фильтр триажа (самый важный): Красный / Жёлтый / Зелёный / Синий.
- Фильтр гео: Доступна ли поликлиника по адресу? (Если нет — предложить ближайшую).

**Каналы** (куда идёт результат):
- Канал 1 (Красный): → API скорой.
- Канал 2 (Жёлтый/Зелёный/Синий): → API расписания → возврат талона.
- Канал 3 (После приёма): → API страховой (это потом, для отчёта).

## 5. Концепция «Цифрового двойника»
*Почему это двойник, а не просто «запись к врачу»?*
Двойник реального регистратора Марины Ивановны, которая:
Смотрит на температуру и говорит «вызывай скорую».
Знает, что бабушку из дома 15 нужно записать в 1-ю поликлинику, а не во 2-ю.
После приёма отправляет отчёт в страховую.
Моя программа эмулирует её знания (правила) и её действия (API вызовы).


### Курсовой 
**Введение** (цель: замена человека-регистратора).
**Функциональные требования** (что должна делать система):
* ФТ-1: Принимать анкету с симптомами.
* ФТ-2: Классифицировать по цветам.
* ФТ-3: Интегрироваться с API МИС.
* ФТ-4: Передавать экстренные вызовы.
**Нефункциональные требования**:
* Время ответа — < 2 секунд (не заставлять пациента ждать).
* Доступность 24/7.
